# Component Spec

```text
Task line: Line B（資訊可信度）
Core model: Site + SiteStatusReport
```

## Problem statement

現場志工在協調站點時，常常收到「有人說某站點可能已經關閉/開放」這類建議，但無法分辨這是已經證實的事實，還是尚待查核的說法。協調者需要一個畫面：能同時看到站點目前的（可能是官方確認的）狀態，以及外部建議，並在建議與已確認狀態衝突時，不被系統自動誤導或覆蓋。

## Users and roles

以下角色是 demo 裡的責任視角，不是登入、權限或帳號系統。

- 原始資訊提供者：現場志工、回報者（透過 `SiteStatusReport` 提出建議狀態）
- 實際操作者：值班協調者（在畫面上決定要「採用建議」「標記需複查」或「駁回建議」）
- 查核者：同為值班協調者（本次 demo 不拆分獨立查核權限系統）
- 資訊接收者：其他志工／團隊（依畫面上顯示的站點狀態決定是否前往）

## In scope

- 顯示所有站點（`Site`）目前狀態、來源、最後確認時間
- 顯示針對該站點的建議狀態更新（`SiteStatusReport`），並清楚標示這只是「建議」
- 已確認（`verified_open` / `verified_closed`）的站點，畫面**不提供**「直接套用建議狀態」的動作，只能「標記需複查」或「駁回建議」
- 非已確認狀態的站點，可以「採用建議」「標記需複查」或「駁回建議」
- 人工動作只影響當前畫面顯示（React state），不寫回 fixture、不使用 localStorage、reload 後回到原始資料
- 沒有待確認建議的站點，清楚顯示「目前沒有待確認的建議」

## Out of scope

- 一個站點同時有多筆建議時的排隊 / 優先順序處理（目前只示範單一建議情境）
- 通知、派工或志工任務指派（屬於 Line C）
- 真實地圖、geocoding、路徑規劃
- 站點狀態變更的歷史紀錄或 audit log 持久化
- 登入、權限系統、角色帳號
- 後端、資料庫、外部 API

## Main user flow

1. 使用者打開「地點」頁籤，看到所有站點列表，含目前狀態 badge、來源、最後確認時間
2. 若某站點有待確認的建議狀態更新，畫面上會標示「有建議待確認」，並顯示建議內容、建議者角色、建議的來源可信度
3. 使用者依目前狀態決定可用動作：
   - 若目前狀態是 `verified_open` / `verified_closed`：只能「標記需複查」或「駁回建議」
   - 其他狀態：可以「採用建議」「標記需複查」或「駁回建議」
4. 動作送出後，畫面即時顯示結果與簡短原因（例如「已標記為需要人工複查，尚未變更為建議狀態」）
5. 若沒有待確認建議，畫面清楚顯示「目前沒有待確認的建議」

## States

| 狀態（`SiteStatus`） | 畫面意義 |
| --- | --- |
| `unknown` | 尚無足夠資訊判斷，需標示「尚未確認」而非留白 |
| `reported_open` / `reported_closed` | 有人回報但未經證實，可被建議覆蓋（採用/複查/駁回皆可） |
| `needs_review` | 已被標記為需要人工複查，等待查核者處理 |
| `verified_open` / `verified_closed` | 已確認，畫面保護：不可被單一建議直接覆蓋 |

## 完成條件（Acceptance Criteria，簡稱 AC）

- [x] AC-1: 使用者可以在畫面上看到所有站點目前的狀態、來源，以及最後確認時間。
- [x] AC-2: 若有人針對某站點提出建議狀態更新，畫面必須清楚標示這是「建議」而非已生效狀態，且不會自動覆蓋原本已確認（`verified_open`/`verified_closed`）的狀態。
- [x] AC-3: 當站點狀態是 `verified_open` / `verified_closed` 時，畫面只提供「標記需複查」與「駁回建議」，不提供「直接套用建議狀態」的按鈕；只有非已確認狀態才能「採用建議」。
- [x] AC-4: 沒有待確認建議的站點，畫面清楚顯示「目前沒有待確認的建議」，不會讓使用者誤以為有事情待處理。

## Failure and uncertain cases

- 建議狀態與目前狀態相同（重複建議）：畫面顯示「建議與目前狀態相同，僅供留意」，不當作新事件處理
- 建議本身來源是 `needs_review` / `unverified`，且指向一個已確認站點：只能標記需複查，不能採用
- 站點狀態是 `unknown` 且沒有任何建議：明確顯示「尚無足夠資訊判斷狀態」，不得留白讓人誤以為已確認開放

## Human confirmation points

- 是否採用建議狀態
- 是否標記為需要人工複查
- 是否駁回建議

以上三個動作都需要人工在畫面上按下按鈕才會發生；系統不會自動判斷站點狀態或建議的真偽。
